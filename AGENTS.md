# AutoCV 工程协作规范

本文档约束本仓库中的人工开发和 AI Agent 开发。产品范围、架构和交付顺序分别以以下文档为准：

- `docs/product/autocv-mvp-product-spec.md`
- `docs/architecture/mvp-architecture.md`
- `docs/plans/mvp-implementation-plan.md`

发生冲突时，先确认产品规格，再确认架构决策，最后按实施计划拆分工作。不要通过代码实现反向扩大 MVP 范围。

## 1. 核心原则

### 1.1 KISS

- 优先选择满足当前需求的最简单方案。
- 不为尚未出现的需求提前设计扩展点。
- 不因为“以后可能需要”引入框架、服务、缓存、队列或复杂抽象。
- 只有在真实代码出现重复、耦合或测试困难时才提取抽象。
- 一个直接函数能够清晰解决的问题，不要先建立多层接口。
- 保持模块职责明确，但不要为了目录结构制造空壳类型。

### 1.2 使用标准能力

- 优先使用语言、框架和工具链已经提供的标准能力。
- 官方 CLI、脚手架、迁移工具、代码生成器或格式化工具能够可靠完成的工作，不手工重复实现。
- 不自行维护框架原本能够生成的绑定、模板代码或配置样板。
- 不用临时字符串处理代替已有的结构化解析 API。
- 引入第三方工具前，先确认标准库或现有依赖是否已经满足需求。

### 1.3 官方文档优先

开始使用或配置框架、SDK、CLI 和库之前：

1. 查看当前版本的官方文档。
2. 确认推荐命令、参数、生成目录和版本要求。
3. 使用 `--help` 或等价命令核对本机安装版本实际支持的参数。
4. 再执行初始化、生成、迁移或构建命令。

资料优先级：

1. 当前版本官方文档。
2. 官方仓库中的 README、示例和迁移指南。
3. 本仓库已经验证的命令和代码。
4. 第三方文章或问答。

第三方资料不能覆盖官方文档和仓库中的已验证事实。命令可能随版本变化时，不凭记忆猜测。

### 1.4 使用 CLI 的边界

- 使用 Wails、Go、Node、数据库迁移和其他工具的官方命令完成初始化与生成。
- 执行生成命令前检查工作区状态，执行后检查生成内容和 `git diff`。
- 生成结果包含无关文件、错误版本或不符合架构的默认实现时，应调整命令或做最小修正，不能不审查地全部接受。
- 不手工编辑明确标注为 generated 的文件；应修改生成源或重新运行生成命令。
- 不为了使用 CLI 而使用 CLI。简单文件编辑比引入新生成器更清楚时，直接编辑。
- 外部命令必须可复现。需要长期使用的命令应进入项目脚本、Makefile、Taskfile 或文档，避免只存在于个人 shell 历史中。

## 2. 开发流程

每个小改动遵循以下循环：

```text
确认范围
  -> 查阅相关官方文档和仓库约束
  -> 使用官方 CLI 或最简单实现
  -> 检查生成或修改内容
  -> 运行针对性验证
  -> 检查 git diff
  -> 提交本次改动
  -> 再开始下一项
```

具体要求：

- 开始前阅读相关模块和现有测试，不猜测代码结构。
- 一次只处理一个可描述、可验证的小功能或小改动。
- 完成当前改动并提交后，再进入下一个独立改动。
- 不把功能、无关重构、格式化和文档更新混成一个大改动。
- 修复问题时先定位根因，再做最小修复。
- 改动公共契约、数据库 Schema 或架构边界时同步更新相关文档或 ADR。
- 发现计划外问题时记录并单独处理，不顺手扩大当前提交。

## 3. 验证要求

提交前至少完成与改动相关的最小验证：

- Go 代码：格式化并运行相关测试；共享逻辑运行 `go test ./...`。
- TypeScript/React：运行类型检查、相关测试和 lint。
- 数据库：运行迁移测试，并验证重复启动或重复迁移。
- AI Schema：使用固定 Fixture 验证成功、格式错误和缺失字段。
- Typst/PDF：至少完成模板编译；影响排版时检查中英文 PDF。
- 文档：检查相对链接、代码围栏和 `git diff --check`。
- Wails 前后端交互：检查生成 Binding，并验证实际桌面流程。

如果某项验证因环境限制无法运行，提交说明和最终交付说明必须明确记录，不能默认视为已经通过。

## 4. Git 提交规范

### 4.1 小步提交

- 一个提交只表达一个完整意图。
- 一个小功能、一个独立修复、一次局部重构或一组紧密相关文档更新分别提交。
- 每个提交应能够独立理解、复盘，并在合理情况下独立回滚。
- 不提交已知无法构建或无法通过对应验证的中间状态。
- 不使用一个大提交覆盖整个里程碑。
- 官方脚手架首次生成的大量关联文件可以作为一个初始化提交，但后续配置、业务代码和功能应继续拆分。

### 4.2 提交时机

满足以下条件后立即提交：

- 当前小改动已经完成。
- 针对性验证已经通过，或限制已经明确记录。
- `git diff` 中没有无关改动。
- 没有密钥、私有简历、真实 JD、数据库、日志或构建产物。

提交完成后再开始下一个小改动。

### 4.3 Commit Message

格式：

```text
<gitmoji> <type>: <简短中文说明>
```

常用类型：

| Gitmoji | Type | 用途 |
| --- | --- | --- |
| ✨ | `feat` | 新功能 |
| 🐛 | `fix` | Bug 修复 |
| ♻️ | `refactor` | 不改变行为的重构 |
| ✅ | `test` | 测试 |
| 📝 | `docs` | 文档 |
| 🔧 | `chore` | 工具、配置和维护 |
| 👷 | `ci` | CI/CD |
| 🎨 | `style` | 不改变逻辑的格式和代码风格 |
| ⚡️ | `perf` | 性能优化 |
| ⬆️ | `build` | 依赖或构建系统升级 |
| 🗃️ | `chore` | 数据库和迁移维护 |

示例：

```text
🔧 chore: 初始化 Wails 工程
🗃️ chore: 添加 SQLite 初始迁移
✨ feat: 支持 Markdown 资料导入
✅ test: 覆盖匹配分计算规则
🐛 fix: 保持锁定内容不被重新生成
📝 docs: 补充本地开发说明
```

要求：

- 说明使用祈使语气，简短描述本次提交完成了什么。
- 不使用“更新代码”“修改文件”“一些调整”等无信息量描述。
- `type` 使用小写英文。
- 标题只描述一个意图；需要解释原因时在提交正文补充。
- 破坏性变更必须在正文中说明影响和迁移方式。

## 5. Git 操作安全

- 提交前运行 `git status` 和 `git diff`。
- 只暂存当前改动涉及的文件。
- 不覆盖、回退或删除他人尚未提交的改动。
- 不使用 `git reset --hard`、强制推送或其他破坏性命令，除非用户明确要求。
- 不为了得到干净工作区而回滚无关变更。
- 发现工作区已有相关改动时，先理解并在其基础上继续。
- 提交后确认 `git status`，确保没有遗漏本次改动或意外暂存文件。

## 6. AutoCV 特定约束

- 当前开发顺序以 M0 到 M5 为准，先完成首个 Markdown 到 PDF 垂直闭环。
- MVP 不引入 Agent 框架、向量数据库、微服务、云账户或完整富文本编辑器。
- Go 持有业务状态、规则和工作流，前端只持有临时界面状态。
- AI 输出必须经过结构化 Schema 和 Go 侧规则校验。
- 未确认量化值不得进入最终简历。
- 用户私有简历、JD、API Key、数据库、日志和导出文件不得提交 Git。
- Job Radar 属于第二阶段，不能进入当前 MVP 提交。

## 7. 文档维护

- 产品行为变化：更新产品规格。
- 技术边界变化：更新架构文档或新增 ADR。
- 交付顺序变化：更新实施计划。
- 开发命令变化：更新 README 或开发文档。
- 工程协作规则变化：更新本文件。

文档更新也遵循小步提交和 Gitmoji 规范。
