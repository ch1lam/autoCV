# AutoCV MVP 产品规格形成计划

> 状态说明（2026-06-10）：本计划保留为完整的样本验证和产品决策清单。项目已经采用“先建立保守的开发基线，再用真实样本迭代验证”的路径，因此 `docs/product/autocv-mvp-product-spec.md`、`docs/architecture/mvp-architecture.md` 和 `docs/plans/mvp-implementation-plan.md` 可以指导 M0 与首个垂直闭环开发。本计划中的高风险真实性规则和 MVP 成功标准仍必须在正式 MVP 发布前完成验证。

## 1. 文档定位

这是一份“计划的计划”：它规定如何收集证据、验证假设、完成产品决策，并最终形成可指导 AutoCV MVP 开发的产品规格。

本文件不是当前产品规格、技术架构或开发任务清单。执行本计划期间，不应因为 `README.md` 或 `docs/architecture/ai-workflow.md` 已经写出某项功能、Agent、数据表或技术方案，就把它视为已经通过真实样本验证的最终要求。

## 2. 最终目标

完成本计划后，应具备把以下开发基线提升为“经过真实样本验证的正式产品规格”所需的全部证据和决策：

```text
docs/product/autocv-mvp-product-spec.md
```

该产品规格应能回答：

- AutoCV 首先为谁解决什么问题。
- 用户如何从本地资料和目标 JD 得到可投递简历。
- MVP 包含和排除哪些能力。
- 系统如何进行经历挖掘、合理推断、包装和事实控制。
- 用户在哪些节点参与确认。
- 什么结果算作 MVP 成功。
- 哪些产品要求将约束后续技术方案。

## 3. 本计划不做什么

- 不编写 PRD 或用户故事。
- 不确定页面、交互和视觉设计。
- 不确定数据库结构、Agent 数量或模块边界。
- 不验证 Wails、Go、TypeScript、SQLite、AI SDK 或 PDF 技术方案。
- 不创建项目脚手架或业务代码。
- 不实现招聘网站监控。
- 不把现有文档中的建议技术栈视为最终结论。

## 4. 当前输入材料

### 4.1 仓库材料

- `README.md`：当前产品愿景、功能范围、MVP 和技术方向草案。
- `docs/architecture/ai-workflow.md`：当前 AI 工作流职责、状态和边界说明。
- 后续由用户提供的真实简历、JD、项目资料和过度包装案例。

### 4.2 已确认的产品方向

以下内容作为本计划的已知约束使用，后续研究负责将其变成可执行定义，而不是重新讨论其是否存在：

- 第一位用户是项目作者本人；验证成功后再扩展到更广泛的求职者。
- 中文和英文简历都需要支持。
- 第一阶段以国内求职场景为主，但不应人为封死海外 JD。
- 产品的两个长期核心价值是针对 JD 定制简历，以及自动发现合适岗位。
- 产品需要通过简历领域规则、资料积累和自动化工作流改善通用 AI 一次性生成效果，减少反复整理上下文和修改提示词的工作，让用户主要负责审阅。
- MVP 先完成简历定制闭环；招聘监控属于第二阶段。
- 用户资料应长期放在本地目录或资料库中，JD 是按机会临时加入的输入。
- 第一版资料输入候选格式为 PDF、Word 和 Markdown。
- 用户可以补充项目文档、工作记录、作品集等资料。
- 资料不足时，系统应主动提问，而不是直接静默补全。
- 第一版不建设完整富文本编辑器；最多允许编辑约定格式的 Markdown。
- 第一版输出 PDF 和 Markdown。
- 允许从已有事实中推断隐含能力，也允许重新解释经历的业务价值。
- 包装强度采用几个明确档位，并同时影响语言力度、关键词、经历取舍、推断程度和职业定位。
- 只有关键亮点需要显示来源，不要求所有文字逐句展示来源。
- 合理推断不需要在最终简历中单独标记，但生成和审阅过程仍需遵守后续批准的真实性规则。
- 用户缺少 JD 要求时，应说明缺口并帮助用户扬长避短。
- 匹配结果需要一个总分。
- 生成成功的最低标准是用户只需少量修改即可投递。
- 简历通常控制在两页内，但不能强制压缩成一页；经历确实较多时可以超过两页。
- 不需要完整展示改写 diff，只需说明本次修改重点。
- 用户可以锁定不允许 AI 改写的内容。
- 真实简历和 3 至 5 份真实 JD 会在测试阶段提供。
- PDF 视觉目标以 `kami` skill 当前可生成的优秀简历效果作为参考。
- 私人样本默认不提交 Git，不要求为本地测试强制脱敏。
- 文件、记录和用户资料保存在本机；不依赖 AutoCV 云端账户；AI 调用只发送必要内容；用户能够导出和删除全部数据。
- 项目采用 MIT 协议，没有既定商业化计划。
- 当前不设置“开源版本必须包含全部核心能力”的额外产品门槛，也不规划开源版与商业版的能力分层。
- 桌面应用是硬性产品形态。
- Wails、Go 和 TypeScript 是强偏好，后续仍需通过产品约束验证适配性。
- 长期应支持多个 AI Provider；第一版可以先支持 OpenAI，并重点研究 Codex 的接入方式。
- MVP 记忆只聚焦用户资料。
- Cover letter、邮件正文和面试准备排除在 MVP 外。
- MVP 首先由项目作者本人完成验收，不以外部用户测试作为首发门槛。

### 4.3 需要通过研究完成定义的事项

以下不是要求用户现在继续回答的问题，而是执行计划时必须产出明确规则的研究项：

- “只需少量修改即可投递”如何被观察和验收。
- 中英文简历是否共用同一套内容结构和质量标准。
- 本地资料库的最小组织方式，以及用户如何更新和选择资料。
- PDF、Word、Markdown 各自允许的内容和排版复杂度。
- 主动追问的触发条件、问题数量和停止条件。
- 包装档位的数量、名称及每档允许改变的内容。
- “关键亮点”的判定规则及来源展示粒度。
- 匹配总分的构成、解释方式和最低可信要求。
- 锁定内容的最小单位和锁定后的改写行为。
- 超过两页简历的具体条件。
- `kami` 代表产品内置渲染能力、外部工作流，还是仅作为质量参考。
- OpenAI/Codex 在第一版中的准确角色和能力边界。

## 5. 最高风险决策

### 5.1 行业惯例量化与真实性

用户允许在缺少原始数字时，依据行业惯例适当补充量化值。这会直接影响产品最重要的可信度边界，因此不能只写成一句提示词规则。

研究必须区分：

- 原始资料明确提供的数字。
- 可以从原始资料计算出的数字。
- 用户能够确认但尚未写入资料的数字。
- 依据上下文推测的合理范围。
- 仅来自行业惯例、无法由用户经历证明的数字。

本计划不预设最终禁止或允许哪一类。进入产品规格前，必须用真实案例比较其简历效果、误导风险和用户审阅成本，并由用户批准最终规则。

### 5.2 合理推断与过度包装

合理推断是 AutoCV 区别于通用模型的核心能力，但当前还缺少可操作边界。研究必须形成足够具体的判断规则，使系统能够挖掘隐藏价值，同时不新增用户没有承担过的职责、没有使用过的技术或无法解释的成果。

### 5.3 “自动”与用户审阅

AutoCV 的价值是减少重复劳动，让用户主要负责审阅。研究需要确定哪些步骤应默认自动执行，哪些内容必须在导出前获得用户确认，避免流程过于繁琐或过度自动。

## 6. 执行阶段

用于正式 MVP 发布验证时，各阶段必须顺序执行。M0 和首个垂直闭环可以按当前开发基线并行推进，但未经验证的结论不得被标记为正式发布规则。

### 阶段 1：现有主张与假设审计

#### 输入

- `README.md`
- `docs/architecture/ai-workflow.md`
- 本计划第 4 节的已确认方向

#### 行动

- 提取两份文档中的产品主张、功能、Agent、技术方案和工程规则。
- 将每一项标为：已确认约束、待验证假设、待定义规则、技术候选或第二阶段能力。
- 找出两份文档与已确认方向之间的冲突。
- 特别检查招聘监控、长期记忆、申请材料和 Agent 数量是否误入 MVP。

#### 产物

```text
docs/discovery/01-assumption-inventory.md
```

#### 验收标准

- 每项主张都能追溯到原文位置或用户确认。
- 产品要求与技术实现建议被明确分开。
- 所有 MVP 范围冲突都有记录。
- 没有通过“已经写在 README 中”证明其正确性。

#### 用户确认门

用户只需确认审计分类是否准确，不重新回答已经确认的问题。

### 阶段 2：首要用户与核心任务建模

#### 输入

- 项目作者当前真实的求职和简历定制流程。
- 目前保存简历、项目资料和 JD 的方式。
- 使用通用 AI 修改简历时最繁琐、效果最差的具体环节。

#### 行动

- 记录从发现岗位到导出投递简历的当前流程。
- 标出重复劳动、判断工作、资料缺口和人工审核点。
- 定义 AutoCV MVP 必须替代的工作，以及仍由用户承担的工作。
- 明确“针对 JD 定制”与“自动发现岗位”两个价值的阶段关系。

#### 产物

```text
docs/discovery/02-primary-user-and-current-workflow.md
```

#### 验收标准

- 目标用户明确为项目作者本人。
- 至少记录一个真实、完整的当前工作流。
- 每个痛点都有实际例子，不只使用“繁琐”或“效果不好”等概括。
- 能说明 AutoCV 相比直接使用通用 AI 多做了什么。
- 不提前设计页面或 Agent。

#### 用户确认门

用户确认当前工作流和核心痛点没有被误解。

### 阶段 3：建立私有验证样本集

#### 输入

- 一份或多份真实中文、英文简历。
- 3 至 5 份真实 JD。
- 可选的项目资料、工作记录和作品集。
- 后续收集的 AI 过度包装案例。
- `kami` 生成的目标 PDF 示例。

#### 行动

- 为每份样本记录语言、岗位、格式、资料完整度和测试目的。
- 至少准备一个资料充分案例、一个资料不足案例和一个容易过度包装的案例。
- 规定本地私有样本目录和 Git 排除要求。
- 不在此阶段对样本执行生成或评分。

#### 产物

```text
docs/discovery/03-sample-corpus-manifest.md
```

私有样本的建议目录仅在执行阶段确认，不由本计划直接创建。

#### 验收标准

- 中文和英文场景均有覆盖。
- PDF、Word、Markdown 至少各有一个解析样本，或明确缺少原因。
- 每个样本都有验证目的。
- 样本包含正常、资料不足和高风险推断场景。
- 私有资料默认不会被 Git 跟踪。

#### 用户确认门

用户确认样本具有代表性，并允许其用于本地分析和测试。

### 阶段 4：端到端任务验证

#### 输入

- 阶段 3 的样本。
- 阶段 2 的真实工作流。

#### 行动

通过人工或受控 AI 辅助方式演练以下任务，但不实现产品：

```text
选择本地用户资料
  -> 加入目标 JD
  -> 分析 JD
  -> 匹配用户经历
  -> 发现资料缺口并提问
  -> 用户补充信息
  -> 按包装档位生成 Markdown 简历
  -> 说明修改重点和匹配结果
  -> 用户审阅并锁定内容
  -> 生成并预览 PDF
```

为每一步记录输入、输出、自动化程度、用户操作、失败状态和恢复方式。

#### 产物

```text
docs/discovery/04-validated-end-to-end-task.md
```

#### 验收标准

- 至少一个真实案例完整走通。
- 明确用户资料先于 JD 长期存在的使用模型。
- 资料不足时确实触发提问。
- 用户不需要在多个工具间手工复制大量中间结果。
- Markdown 和 PDF 都有明确角色。
- 能识别出哪些步骤对“只需审阅”造成过多负担。

#### 用户确认门

用户确认该流程符合自己期望的日常使用方式。

### 阶段 5：内容生成与真实性规则验证

#### 输入

- 阶段 3 的真实样本。
- 阶段 4 中产生的候选简历内容。
- 后续提供的过度包装案例。

#### 行动

- 对直接改写、事实归纳、能力推断、业务价值解释和量化补充分别建立案例。
- 测试包装档位对五个维度的影响：语言力度、关键词、经历取舍、推断程度、职业定位。
- 测试关键亮点来源展示是否足以让用户快速审阅。
- 测试资料不足时主动提问的质量。
- 测试锁定内容在后续重写中是否保持不变。
- 专门验证行业惯例量化值的可接受范围和审阅机制。

#### 产物

```text
docs/discovery/05-content-and-grounding-study.md
docs/decisions/01-content-policy-decision.md
```

#### 验收标准

- 每条规则都有可接受、边界和不可接受案例。
- 包装档位可以通过输出差异被清楚解释。
- 合理推断不会新增不存在的职责或技术。
- 量化补充规则经过真实案例验证，而非只依赖提示词。
- 用户能够快速发现高风险生成内容。
- 关键亮点能够回到来源材料。
- 锁定内容不会被间接改写。

#### 用户确认门

用户必须批准最终的推断、包装、量化和来源展示规则。未批准不得写入产品规格。

### 阶段 6：匹配评价与成功标准验证

#### 输入

- 多份真实 JD 与简历组合。
- 阶段 5 已批准的内容规则。

#### 行动

- 定义匹配总分要表达的含义。
- 验证技能关键词、经历证据、岗位级别、领域经验和缺口等因素。
- 比较分数与文字解释对用户决策的实际帮助。
- 定义“用户只需少量修改即可投递”的观察方法。
- 验证中英文输出、篇幅和 PDF 质量要求。

#### 产物

```text
docs/discovery/06-match-and-quality-study.md
docs/decisions/02-mvp-success-criteria.md
```

#### 验收标准

- 匹配分数具有稳定、可解释的含义。
- 高分不能掩盖缺少证据或事实风险。
- 明确“少量修改”的衡量方式，例如修改轮次、修改内容类型或用户耗时。
- 明确两页以内的默认规则和允许超过两页的条件。
- 中文、英文、Markdown 和 PDF 均有质量检查项。
- 最终标准能够由项目作者本人执行验收。

#### 用户确认门

用户批准匹配分数含义和 MVP 成功标准。

### 阶段 7：MVP 范围收敛

#### 输入

- 阶段 1 至阶段 6 的结论。

#### 行动

- 将能力分为 MVP、第二阶段和暂不规划。
- 判断每个 Agent 名称是否对应真实独立职责，还是只应作为内部工作流步骤。
- 明确完整编辑器、招聘监控、长期行为记忆、申请材料和外部用户测试的范围。
- 为每一项纳入或排除决定提供证据。

#### 产物

```text
docs/decisions/03-mvp-scope.md
```

#### 验收标准

- MVP 能让项目作者从本地资料和目标 JD 获得可投递的 Markdown 与 PDF 简历。
- 招聘网站监控明确属于第二阶段。
- Cover letter、邮件和面试准备明确排除。
- MVP 记忆仅包含用户资料所需能力。
- 完整富文本编辑器不进入 MVP。
- 每项功能都能关联到真实用户任务或明确被排除。
- 有清晰的“不做什么”清单。

#### 用户确认门

用户正式批准 MVP 边界。未批准时可以继续 M0 和首个垂直闭环，但不得把技术基线提升为正式 MVP 发布约束。

### 阶段 8：产品约束驱动的技术可行性研究

#### 输入

- 已批准的 MVP 范围。
- 已批准的隐私、真实性、输入输出和成功标准。

#### 行动

- 验证桌面、本地优先和资料可删除/导出对技术方案的要求。
- 验证当前 Wails、Go、TypeScript 基线是否适合已确认流程。
- 比较 PDF、Word、Markdown 解析方案。
- 明确 `kami` 的可集成方式和替代方案。
- 研究第一版 OpenAI/Codex 支持及未来多 Provider 边界。
- 区分确定性程序逻辑与适合 AI 执行的任务。
- 研究本地资料目录、索引、版本和私有样本隔离要求。

#### 产物

```text
docs/research/01-technical-feasibility.md
docs/decisions/04-technical-constraints.md
```

#### 验收标准

- 每个技术结论都关联到已确认产品要求。
- Wails、Go 和 TypeScript 被验证，而不是因偏好直接通过。
- 第一版 Provider 范围与长期扩展方向明确分开。
- `kami` 的角色有明确结论。
- 本地隐私和 Git 排除要求有可执行方案。
- 不为研究目的扩大业务范围；允许使用 M0/M1 的代码和独立 Spike 作为验证证据。

#### 用户确认门

用户批准技术硬约束和可接受取舍。

### 阶段 9：产品规格就绪审计

#### 输入

- 所有 discovery、decision 和 research 文档。

#### 行动

- 对照本计划逐项核验产物和用户批准记录。

## 8. 全局停止规则

执行过程中遇到以下情况必须暂停询问用户：

- 现有文档与用户已确认方向冲突。
- 某项结论会改变第一阶段与第二阶段边界。
- 某项规则会允许生成用户无法解释或确认的经历。
- 需要将私人样本发送给新的外部服务。
- 需要决定包装、量化或推断的最终边界。
- 需要改变桌面、本地优先、数据可删除/导出等硬约束。
- 需要把新功能加入 MVP。
- 准备从研究进入产品规格编写。

执行过程中不应为以下事项重复询问用户：

- 已经确认的目标用户、语言、主要地区和桌面形态。
- 已经确认的 PDF、Word、Markdown 输入方向。
- 已经确认的 PDF 与 Markdown 输出方向。
- 已经确认的招聘监控第二阶段定位。
- 已经确认排除的申请材料和完整编辑器。

## 9. 计划完成定义

只有同时满足以下条件，才算本计划执行完成：

- 第 7 节列出的全部研究、决策和就绪文档存在并通过各自验收。
- 所有用户确认门都有明确结论。
- 行业惯例量化、合理推断和包装档位有真实样本验证。
- MVP 范围没有混入第二阶段能力。
- 技术约束由产品要求推导，而不是由现有草案倒推。
- 产品规格就绪审计通过。

完成本计划不等于完成 AutoCV MVP。完成后，应根据已批准证据把 `docs/product/autocv-mvp-product-spec.md` 从开发基线更新为正式发布规格。
