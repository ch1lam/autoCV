# AutoCV MVP 产品规格

> 状态：开发基线 v0.1
>
> 日期：2026-06-10
>
> 适用范围：AutoCV 第一阶段简历定制闭环

## 1. 文档目的

本文档定义 AutoCV MVP 必须解决的问题、用户流程、功能边界、内容真实性规则和验收标准。

它是开始 MVP 开发的产品基线，不代表所有产品假设已经完成外部验证。开发过程中可以通过真实简历和 JD 调整细节，但任何调整不得破坏以下硬约束：

- 桌面应用。
- 本地优先。
- 用户资料可导出、可删除。
- AI 只接收当前任务所需内容。
- 生成内容必须能够被用户理解和确认。
- 第一阶段只完成简历定制闭环，招聘监控属于第二阶段。

## 2. 产品定义

AutoCV 是一个本地优先的 AI 简历工作台。

用户维护一份可复用的本地职业资料库。每次提供目标 JD 后，AutoCV 分析岗位要求、匹配资料证据、询问关键缺失信息、生成针对该岗位的 Markdown 简历，并提供 PDF 预览与导出。

MVP 的核心价值不是“帮用户写一份通用简历”，而是把下面这套重复流程变成可复用、可审阅、可追溯的工作流：

```text
本地职业资料
  -> 目标 JD
  -> JD 分析
  -> 经历匹配与缺口识别
  -> 必要追问
  -> 选择包装档位
  -> 生成针对性简历
  -> 审阅、锁定和调整
  -> PDF 预览与导出
```

## 3. 首要用户与使用场景

### 3.1 首要用户

MVP 的首要用户是项目作者本人。

首发不以外部用户测试为门槛，但产品结构不得依赖作者个人目录、固定简历格式或硬编码岗位。

### 3.2 首要任务

用户已经拥有一份或多份简历及项目资料，希望针对具体 JD 快速生成一份可投递版本，同时满足：

- 重要要求被覆盖。
- 强项得到突出。
- 缺口被诚实说明。
- 不生成无法解释的经历、技术或结果。
- 中文和英文均可使用。
- 用户只需审阅和少量修改。

### 3.3 首发平台

- MVP 首先支持 macOS。
- 架构保留 Windows 扩展能力，但 Windows 打包不作为首个闭环的阻塞项。

## 4. 范围

### 4.1 MVP 包含

- 本地职业资料库。
- Markdown、DOCX、文本型 PDF 的资料导入。
- 项目文档、工作记录、作品集等补充资料导入。
- JD 粘贴与本地文本导入。
- JD 结构化分析。
- 资料与 JD 的证据匹配。
- 可解释的总匹配分。
- 缺失要求和弱匹配说明。
- 最多两轮的关键问题追问。
- 中文、英文简历生成。
- 三档包装强度。
- 关键生成亮点的来源追溯。
- 简历内容块锁定。
- 受约束的 Markdown 编辑。
- PDF 预览。
- Markdown 和 PDF 导出。
- 工作流中断恢复。
- 本地数据导出与删除。

### 4.2 MVP 不包含

- 招聘网站自动监控和爬虫。
- 扫描件 OCR。
- Cover letter。
- 申请邮件。
- 面试准备。
- 自动投递。
- 完整富文本编辑器。
- 多人协作。
- 云账户和云同步。
- 基于长期行为的偏好学习。
- 简历模板市场。

## 5. 核心对象

| 对象 | 定义 |
| --- | --- |
| Profile | 用户长期维护的职业资料集合 |
| Source Document | 一份导入的简历、项目文档或工作记录 |
| Evidence | 从来源资料中提取的可复用事实或能力证据 |
| Job Description | 一次求职机会对应的原始 JD |
| Requirement | 从 JD 中提取的岗位要求 |
| Match | Requirement 与 Evidence 之间的关联和强度 |
| Resume Run | 针对一个 JD 执行的一次完整生成工作流 |
| Resume Block | 可独立审阅、追溯和锁定的简历内容块 |
| Source Reference | Resume Block 到 Source Document/Evidence 的引用 |

## 6. 主流程

### 6.1 建立本地资料

1. 用户创建或打开本地 Profile。
2. 用户导入 Markdown、DOCX 或文本型 PDF。
3. 系统提取工作经历、项目、技能、教育、认证和成果。
4. 用户可以查看来源文本和修正基础信息。
5. 系统保存结构化 Evidence，并保留来源位置。

### 6.2 创建 JD 工作区

1. 用户粘贴 JD 或导入本地文本文件。
2. 系统提取岗位、级别、职责、必要技能、加分项、领域信号和约束条件。
3. 用户可以查看原始 JD 和结构化结果。

### 6.3 匹配与追问

1. 系统为每项 Requirement 查找 Evidence。
2. 系统区分强匹配、弱匹配和缺失。
3. 系统计算总匹配分并解释主要得分和扣分来源。
4. 如果补充信息会明显改变简历，系统提出最多 5 个问题。
5. 用户可以回答、跳过或结束追问。

### 6.4 生成与审阅

1. 用户选择语言、包装档位和目标模板。
2. 系统生成结构化简历和 Markdown。
3. 系统执行真实性、JD 覆盖、重复、篇幅和锁定内容检查。
4. 用户查看匹配结果、关键修改说明和来源引用。
5. 用户可以修改 Markdown、锁定 Resume Block 或重新生成未锁定部分。

### 6.5 PDF 与导出

1. 系统把已批准内容渲染成 PDF。
2. 用户在应用内预览 PDF。
3. 超过两页时显示提醒，但不强制阻止导出。
4. 用户导出 Markdown 和 PDF。

## 7. 功能要求

### FR-01 本地 Profile

- 用户可以创建、重命名、打开和删除 Profile。
- 一个 Profile 可以包含多份 Source Document。
- 删除 Profile 必须说明会删除哪些本地数据，并要求二次确认。

### FR-02 文档导入

- Markdown 必须在首个垂直闭环支持。
- DOCX 和文本型 PDF 必须在 MVP 发布前支持。
- 扫描型 PDF 不做 OCR，应提示用户改用可复制文本的文件。
- 导入失败不能损坏已有 Profile。
- 原始文件默认不复制到数据库；系统保存受管理副本或文件引用时必须明确其行为。

### FR-03 资料结构化

- 系统至少提取：基本信息、工作经历、项目、技能、教育、认证和成果。
- 每条 Evidence 必须保留来源文档 ID 和可定位的来源片段。
- 用户修正后的结构化信息优先于后续 AI 重新提取结果。

### FR-04 JD 分析

结构化结果至少包含：

- 岗位名称。
- 岗位级别。
- 主要职责。
- 必要技能。
- 加分技能。
- 领域信号。
- 语言、地点、年限等筛选条件。
- 模糊或冲突信息。

### FR-05 匹配分析

- 每项 Requirement 必须显示匹配状态和依据。
- 匹配状态为 `strong`、`partial`、`missing` 或 `unknown`。
- 系统必须区分“资料中没有写”和“用户明确不具备”。
- 总匹配分必须由结构化 Requirement 和 Evidence 计算，不能直接采用模型输出的主观分数。

### FR-06 追问

满足以下任一条件时可以追问：

- JD 的重要要求存在潜在线索，但证据不足。
- 缺少职责范围、结果、规模、技术细节或业务影响会明显削弱简历。
- 候选量化值需要用户确认。
- 两份来源资料存在冲突。

限制：

- 每轮最多 5 个问题。
- 最多 2 轮。
- 用户可以跳过。
- 跳过后系统必须降低对应表述强度，不能自行补成事实。

### FR-07 包装档位

MVP 固定为三档：

| 档位 | 行为 |
| --- | --- |
| 保守 | 只使用明确事实；主要做选择、排序、压缩和清晰化 |
| 平衡 | 默认档；允许从事实中归纳能力和业务价值，但不新增职责、技术或结果 |
| 强化 | 使用更有竞争力的职业定位和语言力度，扩大相关经历的表达权重，但仍遵守同一真实性边界 |

所有档位都不得：

- 声称用户没有承担过的职责。
- 声称用户没有使用过的技术。
- 把团队成果直接改写成个人独立成果。
- 自动加入未经确认的具体数字。

### FR-08 简历生成

- 输出必须先形成结构化 Resume，再派生 Markdown。
- 同一 Resume Run 的段落和项目必须具有稳定 ID。
- 生成应根据 JD 调整内容选择和顺序，而不是只做关键词替换。
- 默认目标为两页以内。
- 用户经验确实需要时允许超过两页。

### FR-09 来源追溯

- Summary 关键定位、主要成果和高风险推断必须提供 Source Reference。
- 普通过渡语和格式性文字不要求逐句引用。
- 用户应能从简历内容块跳转到来源片段。
- 没有来源且未经用户确认的内容不得进入最终导出。

### FR-10 内容锁定

MVP 锁定单位为 Resume Block，包括：

- Summary。
- 单条工作经历 bullet。
- 单条项目 bullet。
- 单个教育或认证条目。
- 单个技能分组。

重新生成时：

- 锁定块的文本不得变化。
- 允许调整未锁定块。
- 如果锁定块与目标 JD 或篇幅冲突，系统只能提示，不能自动改写。

### FR-11 受约束编辑

- 用户可以编辑生成后的 Markdown。
- 系统必须维护 Markdown 与 Resume Block 的关联。
- 无法解析的结构性修改应提示用户确认，而不是静默丢失锁定和来源信息。

### FR-12 PDF

- PDF 内容必须来自用户当前批准的 Resume 版本。
- PDF 必须支持中英文字符。
- 文本应可选择和复制，满足基础 ATS 可读性。
- 预览和导出结果必须来自同一份渲染产物。
- PDF 渲染失败不能覆盖上一次成功产物。

### FR-13 工作流恢复

- 每个阶段完成后保存结果。
- 应用重启后可以继续未完成的 Resume Run。
- 失败阶段可以重试，已成功阶段默认不重复执行。
- 用户可以从匹配、追问或生成阶段主动重新开始，并明确哪些后续结果会失效。

### FR-14 本地数据控制

- 用户可以导出 Profile 的结构化数据和原始导入清单。
- 用户可以删除 Profile、JD、Resume Run 和导出文件记录。
- API Key 不得明文写入 SQLite、日志或导出包。
- 日志默认不记录完整简历、完整 JD 或模型原始请求。

## 8. 内容真实性规则

### 8.1 信息等级

| 等级 | 定义 | 能否直接写入最终简历 |
| --- | --- | --- |
| A | 来源材料明确提供的事实 | 可以 |
| B | 可由多个来源事实直接计算或归纳 | 可以，保留来源 |
| C | 用户在追问中明确确认的信息 | 可以，记录确认 |
| D | 根据上下文推测的范围或可能性 | 不可以，必须先确认 |
| E | 仅来自行业惯例的数字或经历 | 不可以 |

### 8.2 合理推断

允许：

- 从实际工作内容归纳能力。
- 解释已有工作的业务价值。
- 把分散事实组织成更清晰的成果表达。
- 根据 JD 选择更相关的事实和术语。

不允许：

- 新增不存在的职责、技术、客户或项目。
- 改变用户在项目中的实际角色。
- 把可能性写成确定事实。
- 用行业常见指标替代用户真实指标。

### 8.3 量化

- A、B、C 级数字可以进入最终简历。
- D 级数字只能作为待确认建议展示。
- E 级数字不生成。
- 用户确认数字时，系统记录确认时间和对应 Resume Run。

## 9. 匹配分

### 9.1 计算原则

AI 负责提取 Requirement 和 Evidence，Go 侧使用确定性规则计算分数。

默认权重：

| 维度 | 权重 |
| --- | ---: |
| 必要技能和硬性条件 | 40 |
| 主要职责证据 | 30 |
| 岗位级别与责任范围 | 15 |
| 领域和业务经验 | 10 |
| 加分项 | 5 |

单项证据强度：

- `strong = 1.0`
- `partial = 0.5`
- `missing = 0`
- `unknown = 0`，但必须显示“信息不足”

### 9.2 分数约束

- 存在未满足的明确硬性条件时，总分最高为 69。
- 没有来源证据的模型判断不能增加得分。
- 分数必须同时显示各维度得分、缺失项和不确定项。
- 分数只表示当前资料与 JD 的证据匹配度，不表示获得面试或录用的概率。

## 10. 异常与边界状态

| 场景 | 产品行为 |
| --- | --- |
| AI 未配置 | 允许管理本地资料；进入分析前提示配置 |
| AI 请求失败 | 保存输入和已完成结果，允许重试或更换 Provider |
| 输出不符合 Schema | 自动修复一次，仍失败则显示阶段错误 |
| 文档无法解析 | 保留导入记录并说明原因，不生成空 Evidence |
| JD 太短 | 提示信息不足，允许继续但标记低可信 |
| 来源互相冲突 | 进入追问，不自动选择一个版本 |
| 用户跳过问题 | 降低表述强度并保留缺口 |
| PDF 超过两页 | 显示篇幅警告和可调整建议，允许导出 |
| PDF 渲染失败 | 保留上一次成功版本并显示日志摘要 |

## 11. 非功能要求

### 11.1 隐私

- 用户内容默认只保存在本机。
- AI 请求采用最小上下文原则。
- 发送给 AI 前应显示 Provider 和数据类型摘要。
- 不实现遥测，除非未来单独取得用户同意。

### 11.2 可恢复性

- 应用异常关闭后不丢失已经完成的阶段。
- 数据库写入和文件产物写入必须采用原子替换或事务。

### 11.3 性能

- 打开已有 Profile 的本地界面目标为 2 秒内可操作。
- 不等待 AI 的本地操作目标为 300 毫秒内给出反馈。
- AI 长任务必须显示阶段、状态和取消入口。

### 11.4 可测试性

- 匹配分、锁定规则、状态转换和内容政策必须有单元测试。
- AI 输出通过固定 Schema 和测试样本验证。
- 私有样本不进入 Git；仓库提供合成测试样本。

## 12. MVP 成功标准

使用至少 3 组真实简历与 JD 进行验收，其中至少包含：

- 一组中文场景。
- 一组英文场景。
- 一组资料不足或容易过度包装的场景。

MVP 通过条件：

- 3 组案例都能完成从资料导入到 Markdown/PDF 导出。
- 最终简历不存在用户无法解释或确认的事实。
- 关键亮点能够返回来源或用户确认记录。
- 锁定内容在重新生成后保持不变。
- 匹配分能够解释主要加分和扣分来源。
- 至少 2 组案例在首稿后 15 分钟内完成审阅。
- 至少 2 组案例的人工修改不超过全部 Resume Block 的 20%。
- PDF 中英文可复制、无明显截断，并可正常打开。

## 13. 两个交付门

### 13.1 首个垂直闭环

用于证明架构和核心价值，允许只支持：

- 一个 Profile。
- Markdown 资料导入。
- 粘贴 JD。
- 单一 AI Provider。
- 平衡包装档。
- Markdown 生成。
- 一个 PDF 模板。

但必须完整走通：

```text
Markdown 资料 -> JD -> 分析 -> 匹配 -> 生成 -> PDF 预览与导出
```

### 13.2 MVP 发布

在首个垂直闭环基础上补齐：

- DOCX 和文本型 PDF 导入。
- 三档包装。
- 追问。
- 来源追溯。
- 内容锁定。
- 工作流恢复。
- 数据导出与删除。
- 中文、英文和三组真实样本验收。

## 14. 后续阶段

以下内容在 MVP 稳定后再规划：

- Job Radar 招聘监控。
- 多 Provider 完整支持。
- Windows 正式发布。
- 长期偏好学习。
- 更多简历模板。
- Cover letter、申请邮件和面试准备。
